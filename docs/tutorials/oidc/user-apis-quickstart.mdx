# User APIs Quickstart

Purpose: Start calling User APIs with the recommended architecture.
Use this when: You want bookmarks, collections, preferences, goals, notes, or other signed-in user features.
Do not use this when: You only need Content or Search.
Backend required: Usually yes.
Allowed runtimes: Browser apps, mobile apps, backends.
Required credentials: `client_id` always. `client_secret` only on the backend.
Minimal import: `@quranjs/api/public` in the app and `@quranjs/api/server` on the backend.

## Recommended Setup

- App or mobile client: `@quranjs/api/public`
- Backend: `@quranjs/api/server`

## Frontend Example

```ts
import { createPublicClient } from "@quranjs/api/public";

const client = createPublicClient({
  clientId: "your-client-id",
  clientType: "confidential-proxy",
});
```

## Backend Example

```ts
import { createServerClient } from "@quranjs/api/server";

const client = createServerClient({
  clientId: process.env.CLIENT_ID!,
  clientSecret: process.env.CLIENT_SECRET!,
});
```

## First User API Call

```ts
const userClient = createPublicClient({
  clientId: "your-client-id",
  clientType: "confidential-proxy",
  userSession: {
    accessToken: "user-access-token",
  },
});

const collections = await userClient.auth.v1.collections.list();
```

## Short Flow

1. Frontend creates PKCE.
2. Frontend opens Quran Foundation login.
3. Frontend receives the code.
4. Frontend sends the code to the backend.
5. Backend exchanges the code.
6. Backend stores or refreshes tokens.
7. Frontend or backend calls User APIs with the access token.

If the access token expires, the backend should refresh it with the refresh token before trying again. That is why the backend or BFF is part of the normal setup for confidential clients.

## Common Mistakes

- Trying to call `/oauth2/token` directly from a confidential browser app.
- Using `@quranjs/api/public` for Content or Search.
- Forgetting to pass a user session before calling a User API.
