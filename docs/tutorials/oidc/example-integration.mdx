# Full-Stack OAuth2 Example

Purpose: Show the recommended end-to-end shape for most teams.
Use this when: You have a frontend and backend.
Do not use this when: You have a Quran Foundation-approved public client and no backend.
Backend required: Yes.
Allowed runtimes: Browser apps, mobile apps, backends.
Required credentials: `client_id` in the app, `client_id` and `client_secret` in the backend.
Minimal import: `@quranjs/api/public` and `@quranjs/api/server`.

## Frontend

Start with the official scaffold:

```bash
npx @quranjs/create-app@latest my-quran-app --template next --package-manager npm --install --git --sdk-source npm --yes
```

That starter gives you the fastest production-shaped baseline for external integrations.

Use the standalone `quran-oauth2-sdk-e2e-example` repo as a secondary reference and debug baseline. It is a same-origin React SPA shell on top of an Express backend/BFF, and shows the exact confidential-client flow with server-side token exchange, refresh, logout, authenticated SDK calls, debounced app-authenticated search, optimistic CRUD for notes, bookmarks, and collections, plus QuranReflect profile and feed previews.

That example also demonstrates two separate server-side token paths:

- a user-bound OAuth2 session for user APIs like notes and profile data
- a separate `client_credentials` path with the `content` scope for Quran reads through `client.content.v4.*`

That split is the key idea:

- User APIs depend on the signed-in user's token, so they can fail when that token expires.
- Content and Search depend on a separate app token, so they can keep working even when the user token expires.
- The browser still never sees `client_secret`; the backend keeps it and does the confidential work.

The example keeps user tokens in a server-side session. For serious multi-instance production deployments, use a shared session store such as Redis. If your frontend is deployed at the edge, keep the confidential exchange and refresh flow in a backend or BFF layer instead of moving `client_secret` or refresh-token handling into the edge/browser runtime.

```ts
import { createPublicClient } from "@quranjs/api/public";

const authClient = createPublicClient({
  clientId: "your-client-id",
  clientType: "confidential-proxy",
});

const url = authClient.oauth2.v1.authorizeUrl({
  client_id: "your-client-id",
  redirect_uri: "http://localhost:3000/callback",
  response_type: "code",
  scope:
    "openid offline_access user note collection bookmark post comment sync",
  state: "random-state",
});
```

## Backend

```ts
import { createServerClient } from "@quranjs/api/server";

const apiClient = createServerClient({
  clientId: process.env.CLIENT_ID!,
  clientSecret: process.env.CLIENT_SECRET!,
});

const token = await apiClient.oauth2.v1.exchangeCode({
  code: body.code,
  codeVerifier: body.codeVerifier,
  redirectUri: body.redirectUri,
});
```

## After Login

```ts
const collections = await apiClient.auth.v1.collections.list({
  page: 1,
});
```

Local SDK/template workflows are maintainer-only and intentionally omitted from this public onboarding page.

## Common Mistake

Do not move the backend exchange step into frontend code unless Quran Foundation explicitly marked your client as public.
