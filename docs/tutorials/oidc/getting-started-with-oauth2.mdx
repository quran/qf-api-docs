# OAuth2 Tutorial

Purpose: Explain Quran Foundation OAuth2 in plain English.
Use this when: You need to log users in and call User APIs.
Do not use this when: You only need Content or Search.
Backend required: Usually yes.
Allowed runtimes: Browser apps, mobile apps, backends, workers.
Required credentials: `client_id` always. `client_secret` only for confidential clients.
Minimal import: `@quranjs/api/public` in the app, `@quranjs/api/server` on the backend.

## The Short Answer

- Most Quran Foundation clients are **confidential** by default.
- Confidential clients need a backend for `/oauth2/token`.
- Frontend and mobile apps usually do **PKCE login in the app** and **token exchange on the backend**.
- Only clients that Quran Foundation explicitly marks as **public** can exchange codes directly in the app with PKCE only.

That means there are two token paths:

- The **user token** is for personal actions like notes, bookmarks, collections, and QuranReflect user data.
- The **app token** is for app-level reads like Quran content and search.

Content can still work when user APIs fail because the app token is separate from the user token. The browser never gets the `client_secret`; the backend keeps it and uses it for the confidential token exchange and refresh steps.

## Client Types

| Client type         | Plain-English meaning                            | Can store `client_secret`? | Where token exchange happens |
| ------------------- | ------------------------------------------------ | -------------------------- | ---------------------------- |
| Confidential client | Default client type                              | Yes, backend only          | Backend                      |
| Public client       | Special client type approved by Quran Foundation | No                         | App or browser               |

## Do I Need a Backend?

| What you want to do                                | Backend required? |
| -------------------------------------------------- | ----------------- |
| Log users in with a confidential client            | Yes               |
| Refresh tokens for a confidential client           | Yes               |
| Use a public client that Quran Foundation approved | No                |
| Call Content APIs                                  | Yes               |
| Call Search APIs                                   | Yes               |

## Recommended Flow

1. Frontend or mobile app creates PKCE values.
2. Frontend opens Quran Foundation login.
3. User logs in and approves scopes.
4. Frontend receives the authorization code.
5. Frontend sends `code + code_verifier + redirect_uri` to the backend.
6. Backend exchanges the code with `client_secret`.
7. Backend stores or refreshes tokens.
8. Backend calls Content, Search, or User APIs as needed.

If you deploy the frontend at the edge, keep steps 5 to 8 in a backend or BFF layer. The edge can serve the UI, but confidential exchange, refresh, and session storage still need a safe server-side place.

## Minimal Frontend Example

```ts
import { createPublicClient } from "@quranjs/api/public";

const client = createPublicClient({
  clientId: "your-client-id",
  clientType: "confidential-proxy",
});

const authUrl = client.oauth2.v1.authorizeUrl({
  client_id: "your-client-id",
  redirect_uri: "http://localhost:3000/callback",
  response_type: "code",
  scope: "openid offline_access user bookmark collection",
  state: "random-state",
});
```

## Minimal Backend Example

```ts
import { createServerClient } from "@quranjs/api/server";

const client = createServerClient({
  clientId: process.env.CLIENT_ID!,
  clientSecret: process.env.CLIENT_SECRET!,
});

const token = await client.oauth2.v1.exchangeCode({
  code: "oauth-code",
  codeVerifier: "pkce-verifier",
  redirectUri: "http://localhost:3000/callback",
});
```

## Common Mistakes

- Putting `client_secret` in frontend or mobile code.
- Calling `/oauth2/token` directly from a confidential browser app.
- Treating `id_token` as an API access token.
- Mixing production and prelive hosts.
