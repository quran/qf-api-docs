# Response Examples

Use these examples to design resilient integrations. Endpoint schemas remain the source of truth, but these examples show common product states and error shapes that agents and developers should handle.

## User With An Active Streak

Example for [Get streaks](/docs/user_related_apis_versioned/1.0.0/get-streaks/):

```json
{
  "success": true,
  "data": [
    {
      "id": "streak_active_001",
      "startDate": "2026-06-01",
      "endDate": "2026-06-04",
      "status": "ACTIVE",
      "days": 4
    }
  ],
  "pagination": {
    "startCursor": "cursor_active_start",
    "endCursor": "cursor_active_end",
    "hasNextPage": false,
    "hasPreviousPage": false
  }
}
```

## User With No Current Streak

Example for a successful streak lookup where the user has no matching active streak:

```json
{
  "success": true,
  "data": [],
  "pagination": {
    "startCursor": null,
    "endCursor": null,
    "hasNextPage": false,
    "hasPreviousPage": false
  }
}
```

## Successful 200 With Data

Example for a signed-in User API list endpoint:

```json
{
  "success": true,
  "data": [
    {
      "id": "bookmark_001",
      "verseKey": "1:1",
      "createdAt": "2026-06-04T08:00:00.000Z"
    }
  ],
  "pagination": {
    "startCursor": "bookmark_cursor_start",
    "endCursor": "bookmark_cursor_end",
    "hasNextPage": false,
    "hasPreviousPage": false
  }
}
```

## Successful 200 With Empty Data

Example for a list endpoint when the resource exists but no rows match:

```json
{
  "success": true,
  "data": [],
  "pagination": {
    "startCursor": null,
    "endCursor": null,
    "hasNextPage": false,
    "hasPreviousPage": false
  }
}
```

## Successful 200 With A Single Resource

Example for a read endpoint that returns one object:

```json
{
  "success": true,
  "data": {
    "id": "collection_001",
    "name": "Ramadan reading",
    "isDefault": false
  }
}
```

## Common Error Responses

These examples follow the common error envelope used across generated User API references.

### 400 Invalid Request

```json
{
  "message": "The request is missing required headers or is invalid",
  "type": "invalid_request",
  "success": false
}
```

### 401 Unauthorized

```json
{
  "message": "The request requires user authentication",
  "type": "unauthorized",
  "success": false
}
```

### 403 Forbidden

```json
{
  "message": "The server understood the request, but refuses to authorize it",
  "type": "forbidden",
  "success": false
}
```

### 404 Not Found

```json
{
  "message": "The requested resource could not be found",
  "type": "not_found",
  "success": false
}
```

### 422 Validation Error

```json
{
  "message": "The request was well-formed but was unable to be followed due to semantic errors",
  "type": "unprocessable_entity",
  "success": false
}
```

### 429 Rate Limited

```json
{
  "message": "Too many requests, please try again later",
  "type": "rate_limit_exceeded",
  "success": false
}
```

### 500 Server Error

```json
{
  "message": "The server encountered an internal error and was unable to complete your request",
  "type": "internal_server_error",
  "success": false
}
```

## Integration Guidance

- Treat `success: true` with empty arrays as a valid state, not an error.
- Handle pagination cursors as optional values.
- Do not assume every 200 response contains rows.
- Do not call signed-in User APIs without a valid user session.
- Use `@quranjs/api/server` for signed-in User API calls and keep user tokens server-side.
